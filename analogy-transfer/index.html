醫療安全借用航空業的檢核表、軟體開發借用製造業的精實生產——許多創新不是從零發明，而是把別的領域已被驗證的方法遷移過來。重點不是複製表面形式，而是辨認背後真正有效的結構、機制與因果：抽象化問題 → 找結構相似的領域 → 萃取原理 → 移除外殼 → 重新套用 → 小規模驗證。

借用別人的原理，重新解決自己的問題。

新課題

把問題抽象化

先寫具體困擾，再提升到一般層次——「抽血櫃檯等太久」其實是「需求波動下的有限資源配置問題」。抽象化後，才找得到其他產業的類似解法。

還沒有課題從一個讓你頭痛的具體問題開始。

轉譯表

選擇課題

別問「別人用了什麼工具」，要問「這個工具解決了什麼結構性的問題」。移除產業外殼，只保留功能，才不會僵硬照抄。
還沒有課題先到「課題」分頁建立一個。

最小實驗

小規模驗證

跨域方法不一定能直接套用：選一個流程、用少量案例、設定明確指標、比較導入前後差異，再決定是否擴大。

還沒有可驗證的轉譯先到「轉譯」分頁建立至少一筆。

方法摘要

高品質類比遷移的三重檢核

  • 六步驟：抽象化問題 → 找結構相似的領域 → 萃取底層原理 → 移除產業外殼 → 重新設計 → 最小實驗。
  • 最常見的失敗是只模仿外觀——看到儀表板就做滿圖表、看到檢核表就把所有步驟列成勾選項，沒理解底層問題反而增加負擔。
  • 三重檢核：結構是否相似（相近的限制與風險）、機制是否有效（原方法為什麼能產生結果）、條件是否匹配（人員、文化、資料與資源允許導入）。
  • 真正值得遷移的，不是工具本身，而是工具背後已被驗證的解題邏輯。
